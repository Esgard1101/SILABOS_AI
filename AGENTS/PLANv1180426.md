# Plan de Revisión y Rediseño del Flujo Resultados -> Desempeños -> Habilidades -> Método

## Resumen
- Separar el producto en 2 fases:
  - Fase 1: revisión y actualización de sumillas/bibliografía.
  - Fase 2: generación de sílabos.
- En Fase 2, el flujo quedará así: `Curso -> Bibliografía -> Desempeños y habilidades -> Método -> Calificación -> Confirmación`.
- Los desempeños serán híbridos:
  - Si el curso tiene desempeños oficiales en `performances`, esos mandan.
  - Si no existen, la IA los propone desde `sumilla` + `resultado_aprendizaje`.
- La IA dejará de trabajar solo con categorías y pasará a proponer habilidades exactas desde `skills_catalog`, con validación del docente.
- `teaching_methods` pasará a ser la única fuente de verdad para métodos y deberá influir en cronograma, texto de metodología, técnicas y evaluación sugerida.

## APIs e Interfaces
- `GET /api/methods`: pasa a devolver métodos desde `teaching_methods` con UUID y metadatos completos.
- `GET /api/methods/suggest`: usa métodos de BD y habilidades confirmadas.
- Nuevo endpoint para resolver contexto previo a generación:
  - curso,
  - resultado de aprendizaje,
  - desempeños oficiales o fallback,
  - habilidades sugeridas exactas.
- `POST /api/syllabus/generate-v2`:
  - `teaching_method_id: string | null`
  - `selected_skill_ids: string[]`
  - opcionalmente `performance_source` y `resolved_performances` si se decide persistir explícito.
- El editor y exportación deben leer `metodologia` y método persistidos del JSON, no usar fallback genérico.

## Plan de Pruebas
- Curso con `performances` cargados:
  - el wizard muestra esos desempeños y la IA propone habilidades desde ellos.
- Curso sin `performances`:
  - el backend genera fallback IA y lo marca como tal.
- Método seleccionado desde `teaching_methods`:
  - aparece igual en wizard, generación, editor y exportación.
  - el cronograma sigue `phases/weekly_template`.
  - la sección metodología refleja técnicas y evaluación del método.
- Habilidades exactas:
  - se sugieren desde el catálogo de 300,
  - se pueden confirmar o ajustar,
  - se usan luego para logros, habilidades requeridas e instrumentos.
- Compatibilidad:
  - sílabos viejos siguen abriendo en editor aunque no tengan `metodologia` o método persistido.
- Permisos de sumillas:
  - docente responsable edita,
  - coordinador/director supervisan según asignación,
  - usuarios sin asignación no acceden.

## Supuestos y Decisiones
- `teaching_methods` ya existe y será la fuente oficial; el catálogo hardcodeado se elimina del flujo funcional.
- El número inicial de habilidades sugeridas por IA será corto y operable en UI, por defecto 3 a 5.
- Las categorías de `skills_catalog` quedarán como metadato de apoyo/filtro, no como paso principal del wizard.
- No se unifica todavía sumillas y sílabos en una sola pantalla; se mantienen como fases separadas.
- Los cambios de esquema necesarios se propondrán primero como SQL manual para Supabase antes de implementar código.
